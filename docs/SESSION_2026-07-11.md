# Session log — 2026-07-11

A narrative record of this working session on SentinelAI: what was asked, in what order, the
decisions made along the way, and how the work unfolded. This is a companion to
**[PROGRESS.md](PROGRESS.md)**, which is the enhancement-by-enhancement technical changelog — this
file is the conversational "how we got here," not a duplicate of the technical detail already
written there.

## Context at the start

The session picked up mid-project: SentinelAI already had a working NestJS + Next.js stack, and the
user had previously handed down a large, structured 12-phase master plan (landing page, dashboard,
onboarding, security scoring, AI advisor, executive reports, alert center, trust pages, pricing,
customer acquisition, SEO, production review) with the explicit instruction to work through it
feature by feature — review, test, refactor, only then continue — without pausing to ask permission
between phases. Phases 1–8 were already complete going into this session; this session picked up
partway through, continuing autonomously per that standing instruction.

A second, much larger "master AI engineering document" arrived unsolicited partway through,
proposing RAG/vector-DB knowledge systems, local LLM serving (Ollama/vLLM), a full Prometheus/
Grafana/OpenTelemetry observability stack, a public API with JS/Python SDKs, an integrations
marketplace, and white-label support for MSPs. This was flagged back to the user as premature
scaling for a product with no real users yet, no revenue, and inert third-party credentials
(Stripe/AI/SMTP) — the recommendation was to keep working through the practical 12-phase plan
instead, and the user didn't object, so that's what continued.

## Phase 9 — Pricing page overhaul

Added an Enterprise (contact-sales) tier and a real monthly/yearly billing toggle, wired through to
the backend's Stripe integration (separate env vars for yearly Price objects, honest
`BillingNotConfiguredError` if a yearly price isn't configured rather than silently charging the
monthly rate). Verified live in English and Hebrew.

## Phase 10 — Customer acquisition (free-scan lead gen)

Built a real, unauthenticated "free instant scan" widget on the landing page — genuine DNS/TLS/HTTP
probes through the same SSRF-guarded services the authenticated product uses, rate-limited, nothing
persisted for anonymous visitors. Refactored the scoring formula into a shared `scoring.util.ts` so
the anonymous teaser and the real authenticated scan could never diverge on the same signals — this
was the same class of bug (duplicated scoring logic) found and fixed multiple times earlier in the
project's history, so this time it was fixed at the source instead of copied a fifth time.

## Phase 11 — SEO

Found that every public page shared the homepage's `<title>`/meta description (a real, user-visible
bug — search results and browser tabs showed the wrong text for every page except the homepage).
Also found the hreflang tags were wrong for every page except the homepage (pointing at the locale
root instead of the equivalent page). Fixed both with a shared `buildMetadata()` helper, added real
JSON-LD structured data (Organization + FAQPage, generated from the same content actually shown on
the page, not a separate copy).

## Phase 12 — Production review

A genuine pre-launch audit rather than new feature work, per the master plan's own "continue only
after everything is production ready" rule. Found two real, concrete gaps:

- The frontend served zero security headers, despite the `/security` page's own text claiming
  otherwise (true only for the backend's API responses, via helmet — never true for the Next.js
  app's own HTML). Added a matching CSP/HSTS/X-Frame-Options config.
- `next.config.ts` had `output: "standalone"` set (for a smaller Docker image) but the `start`
  script was still plain `next start`, which doesn't correctly serve static assets against a
  standalone build — a real deploy following the obvious `npm run build && npm start` would have
  404'd every CSS/JS/image asset. Fixed the script to copy `public/`/`.next/static` into the
  standalone output first.

Also audited dependencies, `.env.example` completeness, and confirmed Prisma migration state was
clean.

After the 12 phases were reported done, the user re-sent the large master-scope document a second

was framed back to the user as a prioritization choice — pick one real, scoped next step instead of

Partway through this exchange the user also asked to switch to Hebrew for the rest of the
conversation, which has applied from that point onward.

## Two-factor authentication (TOTP)

Built real TOTP-based 2FA: encrypted secrets at rest (AES-256-GCM, a separate concern from password
hashing since the secret must be *decryptable* to verify codes), a genuine two-step enable
commitment (a secret isn't "protecting the account" until the user proves they can generate a real
code from it), single-use argon2-hashed backup codes, and a separately-signed short-lived challenge
token between password verification and the second factor (deliberately a different signing secret
than real access tokens, so it can never be replayed as one).

Two real bugs surfaced during implementation, not before:

1. `otplib` v13 pulled in an ESM-only dependency with no CommonJS build at all — invisible in the
   real running app (built via `nest build`), but it silently broke **every** e2e test, not just
   the new MFA one, the moment the service was imported. Replaced the library entirely with a
   small, self-contained RFC 6238 TOTP implementation built only on Node's own `crypto.createHmac`.
2. Live browser testing (not just automated tests) caught a UX-breaking bug the test suite hadn't:
   entering a wrong or reused MFA code silently reset the whole login page back to an empty form
   instead of showing an error. Root cause was the global 401-response interceptor treating a wrong
   MFA code exactly like an expired session. Fixed and covered with a regression test.

## A cost question, then a regulatory one

The user asked what's left for them personally to do to actually launch (real accounts/API keys
needed, domain + hosting, business/legal registration) — answered as a prioritized checklist, since
these are all things only the user can act on. A follow-up question about rough costs was answered
with real, sourced-from-general-knowledge price ranges (hosting, SMTP, Stripe/Coinbase per-
transaction fees, Anthropic usage-based pricing, business registration, legal review), explicitly
flagged as estimates rather than guarantees.

The user then asked specifically about Israeli regulatory accessibility requirements — a real legal
question (Israeli Standard 5568 / WCAG 2.0 AA, under the Equal Rights for Persons with Disabilities
Regulations, with genuine lawsuit exposure including statutory damages without proof of harm) —
answered with the general legal framework (flagged as not legal advice, recommending a real
accessibility consultant before relying on it commercially), followed by actual engineering work
rather than a cosmetic accessibility widget.

## Accessibility remediation

A background research agent audited the real codebase for concrete WCAG violations before any code
was touched. Findings, all fixed:

- A systemic color-contrast failure: `text-gray-500` (used 101 times across the app) measured
  4.16:1 against the dark background, below the 4.5:1 AA requirement — computed with real WCAG
  luminance math, not estimated, and fixed by collapsing it (and the even-worse-failing
  `text-gray-600`) into `text-gray-400` everywhere.
- No "skip to main content" link existed anywhere in the app — added one, plus `id="main-content"`
  on all seven real `<main>` elements site-wide.
- Zero `<label htmlFor>`/`id` associations existed anywhere in the codebase — every visibly-labeled
  form field was, to a screen reader, unlabeled. Fixed across every form in the app, plus several
  inputs that had relied on `placeholder` text alone with no label at all.
- A heading-hierarchy skip (`h1` straight to `h3`) on the dashboard's real content.
- A keyboard-inaccessible mobile drawer backdrop (a bare `<div onClick>` with no keyboard handler),
  replaced with a real, labeled `<button>`.
- Several inputs with a focus style too subtle for low-vision keyboard users, standardized to match
  the ring-based focus style already used elsewhere.

Added a real, honest `/accessibility` statement page — deliberately not claiming full certified
Israeli Standard 5568 compliance (that would itself be a false, liability-creating claim), stating
plainly what's actually implemented and that a professional third-party audit hasn't happened yet.

Every fix in this session was verified live in a real browser (Playwright), not just by passing
tests — including tabbing to the skip link, confirming Playwright's own accessibility tree now
reports real accessible names for form fields that previously had none, and confirming the mobile
drawer closes on a real keyboard `Enter` press.
